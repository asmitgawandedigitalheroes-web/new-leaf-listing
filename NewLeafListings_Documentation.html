Section One
System
Overview
A complete introduction to the New Leaf Listings platform — its purpose, core capabilities, target users, and high-level operational metrics.
New Leaf Listings is a full-stack real estate SaaS platform that connects property buyers and sellers with licensed realtors operating under regional directors. It provides structured listing management, intelligent lead routing, subscription-based access tiers, commission tracking, and multi-CRM synchronisation within a single unified interface.
Reader's map: This section covers the platform's mission statement, key feature set, and capacity benchmarks. For architecture internals proceed to Section 02; for the database schema see Section 04.
Scope of This Document
This handbook is the authoritative technical reference for engineers, product owners, and onboarding administrators. It documents every layer of the stack — from React hooks and service classes through Supabase RLS policies to Stripe webhook processing — and serves as the sign-off artefact for formal platform acceptance.

1.1 — Platform Purpose & Key Features
Platform Purpose
New Leaf Listings was built to address three persistent pain points in regional real estate operations: fragmented listing management, opaque lead attribution, and manual commission reconciliation. The platform unifies these into a single SaaS product with three distinct role tiers — Admin, Director, and Realtor — each with scoped permissions and dashboards.
The platform is designed for a franchise or territory-based real estate agency model: a national or regional administrator oversees multiple geographic territories, each managed by a Director who recruits and supervises Realtors. Buyer and seller leads flow in from public listing pages and contact forms, are automatically routed to the best-matched Realtor, and are tracked through to conversion with full audit trails.
1.2 Key Features at a Glance
Listing Management
Full CRUD for property listings with a 5-stage approval pipeline (draft → pending → active → sold/expired). Supports image galleries, property details, and three upgrade tiers.
Intelligent Lead Routing
Scoring engine assigns new leads to the highest-ranked available Realtor using subscription tier, listing upgrades, territory sponsorship, and current lead load as inputs.
Commission Tracking
Multi-recipient commission records (Realtor 25%, Director 15%, Sponsor 10%) with a pending → approved → payable → paid status workflow and admin override.
Subscription Billing
Four Stripe-backed plans (Starter $9, Pro $29, Dominator $79, Sponsor $199) with per-plan feature gating enforced at hook level using usePlanAccess.
CRM Synchronisation
Webhook-based sync to GoHighLevel and other providers. Mock mode for dev, retry queue with exponential backoff (5 min → 15 min → 45 min) for production failures.
Audit & Compliance
Every significant action (lead assignment, listing approval, commission change, user invite) is recorded in audit_logs with actor, entity, and metadata.
Supported Property Types
Residential Commercial Land & Lots Multi-Family Industrial Condos & Townhomes Vacation & Short-Term
Platform Access Channels
Public Web Portal
Browse listings, view map, submit contact enquiries, and read pricing — all accessible without authentication. SEO-optimised listing detail pages at /listing/:id.
Authenticated Dashboard
Role-scoped dashboards for Admin (/admin/*), Director (/director/*), and Realtor (/realtor/*) accessed after Supabase Auth login.

Section Two
Architecture &
Tech Stack
A detailed breakdown of the frontend, backend, and infrastructure layers — including dependency versions, service boundaries, and the end-to-end data flow diagram.
The New Leaf Listings platform follows a client-driven SaaS architecture: a React single-page application communicates directly with Supabase (PostgreSQL + Auth + Edge Functions) for all data operations, while Stripe handles payment processing and webhooks are consumed by serverless Edge Functions. There is no traditional Node.js middleware layer — business logic lives in React hooks, TypeScript service modules, and Supabase RLS policies.
Design principle: All sensitive operations (user creation, Stripe webhooks, commission approvals) are handled server-side via Supabase Edge Functions using the service-role key, ensuring the browser never has elevated privileges.
Layer Summary
Presentation
React 19 SPA served via Vite dev server or static CDN build. Tailwind CSS for styling. React Router DOM v7 for client-side navigation.
Data & Auth
Supabase hosted PostgreSQL with Row-Level Security. Supabase Auth (JWT, email/password). Supabase Edge Functions (Deno runtime) for privileged operations.

Backend & Database Layer
All persistent data lives in a Supabase-hosted PostgreSQL instance. The application communicates with it through the Supabase JavaScript client initialised in src/lib/supabase.js using the project's public anon key. Row-Level Security policies enforce data isolation at the database layer so no privileged proxy server is required for normal reads and writes.

User Roles &
Permissions
Detailed breakdown of the three-tier role system — Admin, Director, and Realtor — including the RBAC permissions matrix, plan-based feature gating, and the onboarding approval flow.
New Leaf Listings uses a hierarchical role model. A single Admin controls the entire platform and all territories. Each Director owns one territory and manages the Realtors within it. Realtors are the front-line agents who create listings, manage leads, and earn commissions. Role is stored in the profiles.role column and enforced at three levels: React route guards, Supabase RLS policies, and the useRBAC hook.
Security note: Role assignment is never self-service. New signups default to realtor with status = pending until an Admin approves them. Directors are created directly by Admins via the Add User page.
Onboarding Flow Summary
1
Realtor signs up via /signup. Profile created with role = realtor, status = pending.
2
Redirected to /onboarding/pending — shown a waiting screen.
3
Admin reviews the Approvals Queue at /admin/approvals and approves or rejects.
4
On approval, status changes to active; Realtor gains dashboard access.
5
Director accounts are created by Admin via /admin/add-user; invite link sent via Edge Function.

Database
Schema
Complete PostgreSQL schema reference — all 18 tables, their columns, foreign keys, enum types, RLS policies, and performance indexes.
The entire platform state is stored in a single Supabase-hosted PostgreSQL database. Row-Level Security (RLS) is enabled on every user-facing table, enforcing data isolation without a middleware proxy. The schema is defined in supabase/schema.sql and evolved via incremental migration files in supabase/migrations_*.sql.
Schema design note: All primary keys are UUIDs generated with gen_random_uuid(). All tables include created_at TIMESTAMPTZ DEFAULT now(). Monetary values are stored as NUMERIC(10,2) in USD.

Lead Routing Engine
routing.service.ts implements the platform's intelligent lead distribution algorithm. When a new lead is created without an explicit Realtor assignment, routeLead(context) scores every eligible Realtor in the matching territory and assigns the lead to the highest scorer. The entire decision is logged to audit_logs for traceability.
routeLead() Flow
1
Detect territory — calls territory.service.detectTerritory(location) to find the matching territory by city/state from the lead's enquiry data.
2
Fetch eligible Realtors — queries profiles in that territory with status = active. Loads each Realtor's subscription plan, active listing upgrades, and current open lead count.
3
Score each Realtor — applies the four-factor scoring model (subscription tier, listing upgrade, sponsor bonus, lead load penalty). See score table below.
4
Assign to winner — updates leads.assigned_realtor_id and sets attribution_expiry = now() + 180 days, lock_until = now() + 180 days.
5
Fallback — if no eligible Realtors exist, lead is assigned to the territory Director (assigned_director_id) for manual routing.
6

8.2 — 180-Day Attribution Lock
180-Day Attribution Lock
The attribution lock is the platform's primary fair-compensation mechanism. When a lead is assigned to a Realtor, two timestamps are written to the leads table: attribution_expiry and lock_until — both set to 180 days from assignment time. During this period, no other Realtor can be assigned the lead, ensuring the original Realtor receives commission credit if the deal closes within the window.

Lock Enforcement Points
Database Level
RLS UPDATE policy on leads checks lock_until <= now() before allowing assigned_realtor_id changes. Service-role bypass only via explicit lock-release RPC.
Service Level
lead.service.assignLead() reads lock_until before writing. Throws a typed error if the lock is active, surfaced as a toast in the UI.
UI Level
Lead Drawer shows a lock badge with days remaining. Reassign controls are disabled and grayed out. A "Release Lock" button visible to Directors and Admins only.
Lock Release Workflow
1
Director or Admin opens Lead Drawer for a locked lead. Lock badge shows days remaining in amber.
2

3
reassignLead() called with force: true — sets lock_until = now() and logs to audit (lead.lock_released action).
4
